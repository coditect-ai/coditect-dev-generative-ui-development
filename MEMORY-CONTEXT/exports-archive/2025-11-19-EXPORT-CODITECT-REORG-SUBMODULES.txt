> is this renaming going to cause a lot of code refactoring? 

⏺ Great practical question. Let me assess the actual impact:

  What Renaming Affects

  Low/No Impact (No Code Changes)

  - Internal code - Imports use relative paths, not repo names
  - Submodule references - Update .gitmodules, run git submodule sync
  - Local development - Just re-clone or update remotes

  Medium Impact (Config/Doc Updates)

  - .gitmodules - Update all URLs (automated, one file)
  - READMEs - GitHub links need updating
  - CI/CD workflows - Repo name references in GitHub Actions
  - CLAUDE.md files - Any hardcoded repo URLs

  Higher Impact (Needs Careful Handling)

  - install.sh / update.sh - Uses
  raw.githubusercontent.com/halcasteel/coditect/...
  - Published packages - If any repos are npm/PyPI packages
  - External documentation - Customer-facing links

  Good News

  GitHub redirects help: When you rename a repo, GitHub automatically
  redirects the old URL to the new one. This means:
  - Existing clones still work
  - Old links don't immediately break
  - You have time to update references

  Estimated Effort

  | Task                   | Time    | Automation         |
  |------------------------|---------|--------------------|
  | Rename repos on GitHub | 30 min  | Manual (GitHub UI) |
  | Update .gitmodules     | 10 min  | Script             |
  | Update install.sh URLs | 15 min  | Manual             |
  | Update READMEs/docs    | 1-2 hrs | Search & replace   |
  | Update CI workflows    | 30 min  | Grep & edit        |

  Total: ~3-4 hours of config/doc updates, not code refactoring.

  The Real Question

  Do you want to:
  1. Rename now - Clean slate, 3-4 hours of updates, done forever
  2. Rename incrementally - Do a few at a time as you touch them
  3. Keep names, reorganize structure - Use GitHub topics/folders instead

  What's your preference?
